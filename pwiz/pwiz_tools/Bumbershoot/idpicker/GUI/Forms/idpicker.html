Usage

a)    Importing raw peptide identifications into the IDPicker pipeline

IDPicker currently supports reading raw peptide identifications via the Seattle Proteome Centers pepXML format.  Many search engines have output that can be converted to this format using various SPC tools.  But the pepXML format is not convenient to store the protein assembly data that is generated and used in the IDPicker pipeline.  Instead, raw peptide identifications are explicitly converted to their corresponding false discovery rates (FDR) based on validation via a target-decoy search strategy.  During the conversion process, the FASTA protein database (which must be the same between all files given as input to Qonvert) which was used for the search is read in order to combine the proteins that map with confidently identified peptides to their respective descriptions from the FASTA database.  The database is also used to find an offset into the protein for each peptide identification.  If a peptide maps to more than one protein in the database, the offset is calculated using the first protein only.  Typically, some amount of FDR filtering is done at this step to increase processing efficiency and reduce file size, but it is not required.  Basic usage of IDPickerQonvert is easy, with most of the settings defaulting to typical values:

idpQonvert

[<pepXML filemask> [<another pepXML filemask>] ...] [-b <filepath to list of input pepXML filemasks, one per line>] [-dump] [-cfg <config filepath>]

 

Input pepXML files can be specified as filemasks (i.e. supporting wildcards) on the command line, or they can be read from a batch file which lists each input filemask on a separate line.  The default filtering and validation parameters can be overridden by using the -Name <value> notation on the command line, e.g. -MaxAbsoluteFDR 0.05; these overrides can occur anywhere on the command line.  The parameters can also be set via a configuration file: idpQonvert will look for idpQonvert.cfg in the working directory and use the settings there if it exists, or you can pass a filepath to a custom configuration file with the -cfg flag.  The runtime parameters specific to IDPickerQonvert are as follows:

 

Name

Type (Default)

Description

MaxResultRank

integer (1)

This parameter sets the maximum rank that a search result can have to be passed through to the qonverted file.  Things get really messy when one starts accepting more than the highest ranked hit, so the default value here is usually best.  Note: when peptide candidates get equal scores, they all belong to the same rank, so it is possible to have more than one peptide in a rank 1 result.

MaxAbsoluteFDR

real (0.25)

This parameter sets the maximum FDR that the xcorr-type score of each peptide identification must meet (e.g. 0.25 for 75% confidence or 25% FDR).

MaxRelativeFDR

real (1.0)

This parameter sets the maximum confidence that the deltcn-type score of each peptide identification must meet (e.g. 0.25 for 75% confidence or 25% FDR).  Note: the two FDR filters have an AND relationship - a peptide must pass both filters to be accepted.

ProteinDatabase

string (automatic)

By default, IDPickerQonverter will try to use the database found in the input pepXML files, but if none of those can be found, this parameter will need to be set to a valid filepath to the same database which was used in the input data.  If this parameter is set, it overrides the databases stored in the pepXML files.

DecoyRatio

real (1.0)

If the number of target and decoy proteins in the database is not equal, this parameter must be set to describe the ratio of target proteins to decoy proteins in the database (equal to # of target proteins divided by # of decoy proteins).  This is not done automatically for reasons of efficiency.

DecoyPrefix

string (rev_)

IDPickerQonvert determines if a protein accession ID is a decoy or not by a certain unique prefix.  This parameter sets that prefix.

 

Example usage (with files source1.pepXML, source2.pepXML, source3.pepXML, all searched against database c:\proteindb-with-reverse.fasta):

>        idpQonvert source1.pepXML source2.pepXML source3.pepXML

|           Reading protein database from filepath: "c:\proteindb-with-reverse.fasta"

|           Reading pepXML file from filepath: "source1. pepXML"

|           Writing validated identifications to filepath: "source1.xml"

|           Reading pepXML file from filepath: "source2. pepXML"

|           Writing validated identifications to filepath: "source2.xml"

|           Reading pepXML file from filepath: "source3. pepXML"

|           Writing validated identifications to filepath: "source3.xml"

 

 

b)    Assembling and grouping identifications into an experimental hierarchy

IDPicker supports protein assembly and analysis with arbitrarily complex hierarchies.  This important information is not lost when analyzing the data.  A few relatively simple use cases could be an IDPicker analysis on a single LC-MS run, a group of LC-MS runs (like a MudPIT), or a group of groups of LC-MS runs (a group of MudPITs).  A more complex use case could be a multi-level hierarchy across multiple instrument types, search engines, replicate types, etc.  This information is injected into the analysis during this step of the pipeline (while at the same time consolidating identification data from multiple sources into a single file).  The final confidence filtering level is usually used at this stage, but that need not always be the case.  Basic usage of IDPickerAssemble is easy, with most of the settings defaulting to typical values:

idpAssemble

<output filepath> [<source group name> <source filemask> [<another group name> <another source filemask>] ...] [-b <filepath to list of group/filemask pairs, one per line>] [-dump] [-cfg <config filepath>]

 

The first required argument is the filepath to which the assembled identifications will be written, e.g. c:\assembled-ids.xml.  The other required parameters are given as group/filemask pairs, where the filemask represents one or more qonverted files and the group describes where those files belong in the experimental hierarchy.  Groups are described by a POSIX-like path, with / being the root group and any other group path, e.g. /some/sub/groups, creates subgroups within the hierarchy (similar to a computer filesystem).  The group names can contain spaces, but the path will need to be quoted for it to be parsed correctly.  Like in IDPickerQonvert, the default filtering and validation parameters can be overridden by using the -Name <value> notation on the command line, e.g. -MaxAbsoluteFDR 0.05; these overrides can occur anywhere on the command line.  The parameters can also be set via a configuration file: idpAssemble will look for idpAssemble.cfg in the working directory and use the settings there if it exists, or you can pass a filepath to a custom configuration file with the -cfg flag.  The runtime parameters specific to IDPickerAssemble are as follows:

 

Name

Type (Default)

Description

MaxResultRank

integer (1)

This parameter sets the maximum rank that a search result can have to be passed through to the assembled file.

MaxAbsoluteFDR

real (0.05)

This parameter sets the maximum FDR that the xcorr-type score of each peptide identification must meet (e.g. 0.25 for 75% confidence or 25% FDR).

MaxRelativeFDR

real (1.0)

This parameter sets the maximum confidence that the deltcn-type score of each peptide identification must meet (e.g. 0.25 for 75% confidence or 25% FDR).  Note: the two FDR filters have an AND relationship - a peptide must pass both filters to be accepted.

c)     Generating an IDPicker HTML report

The final stage of the IDPicker analysis pipeline applies the final filters to a file assembled in the previous stage and creates an HTML report in a new directory (the name of the output directory is one of the command line parameters).  Basic usage of IDPickerReport is easy, with most of the settings defaulting to typical values:

idpReport

<output name> <input peptides XML filemask> [<another filemask> ...] [-dump] [-cfg <config filepath>]

 

The first required argument is a path to a directory (which may or may not exist yet) to which the files for the HTML report will be written, e.g. reports/TestReport2000.  The other required parameters are simply filemasks representing one or more qonverted IDPicker files.  Like in IDPickerQonvert, the default filtering and validation parameters can be overridden by using the -Name <value> notation on the command line, e.g. -MaxAbsoluteFDR 0.05; these overrides can occur anywhere on the command line.  The parameters can also be set via a configuration file: idpReport will look for idpReport.cfg in the working directory and use the settings there if it exists, or you can pass a filepath to a custom configuration file with the -cfg flag.  The runtime parameters specific to IDPickerReport are as follows:

 

Name

Type (Default)

Description

MaxResultRank

integer (1)

This parameter sets the maximum rank that a search result can have to be passed through to the assembled file.

MaxAbsoluteFDR

real (0.05)

This parameter sets the maximum FDR that the xcorr-type score of each peptide identification must meet (e.g. 0.25 for 75% confidence or 25% FDR).

MaxRelativeFDR

real (1.0)

This parameter sets the maximum confidence that the deltcn-type score of each peptide identification must meet (e.g. 0.25 for 75% confidence or 25% FDR).  Note: the two FDR filters have an AND relationship - a peptide must pass both filters to be accepted.

MinDistinctPeptides

integer (2)

A distinct peptide is a subsequence of a proteins amino acid sequence, which has been identified (and passed filters) at least once based on the given input, but may be identified more than once or thousands of times.  A thousand identifications of the same subsequence still counts as one distinct peptide, and this parameter sets the minimum number of distinct peptides that a protein must be linked to in order to be included in the final report.

MaxAmbiguousIds

integer (2)

An ambiguous identification comes from a result that has multiple peptides with equal scores, i.e. belonging to the same rank.  Such identifications can severely distort the efficacy of IDPickers connected component (clustering) analysis, so this filter can be set to reject them outright to prevent it.

MostParsimoniousAnalysis

Boolean (true)

This parameter controls whether the final protein list is filtered by the minimum covering set, i.e. the minimum set of proteins that is necessary to describe all identified peptides.  This kind of filter is known as parsimony or Occams Razor.

GenerateBipartiteGraphs

Boolean (false)

A sometimes useful feature of IDPickerReport is the ability to generate GraphViz representations of the connected components (clusters) in the workspace.  Enabling this feature requires that GraphVizs dot program be accessible via the default system path.

RawSourceHostURL

string (http://localhost/)

When IDPicker reports are generated with the intent to be viewed from a web server, this setting describes the host URL to use for the server-side interactive content in the report (e.g. the on-the-fly spectrum viewer via the generateSpectrumSvg CGI program).

RawSourceExtension

string (.xml)

This parameter sets the extension that a spectrum source file is expected to be (generateSpectrumSvg requires the mzData format).

RawSourcePath

string ()

This parameter sets the path to the spectrum source files on the web server.  If it is left at the default empty string, no server-side interactive content is generated in the HTML report.

Glossary (a.k.a. what do all these ambiguous terms actually mean?!)

Click on a term to look it up in the Wikipedia.

 

/

The root source group.  This group always contains the analysis results for all other source groups combined.

Bipartite Graph

A visual way of representing the relationships between protein groups and peptide groups.

CID

A unique cluster identification number for a given analysis.

Cluster

A set of interrelated protein groups and peptide groups.  A cluster cannot be interrelated with any other cluster.  In graph theory, this concept is called a connected component.

 

False Discovery Rate (FDR)

A statistical estimate of the probability that a given result is a false one.  The absolute FDR is based on the absolute scores that the search engines supply (e.g. XCorr, MVH, Hyperscore).  The relative FDR is based on the corresponding relative scores (e.g. DeltCn, Expectation Value).

GID

A unique global identification number for either a protein group or a peptide group.

GLID

A numeric or alphabetic symbol used as a cluster-local identifier for a protein group or peptide group.

Identification (id)

A match between a spectrum and a peptide with assigned false discovery rates.

Minimum Covering Set (MCS)

The smallest set of protein groups necessary to explain the existence of all peptide groups in a cluster.

Peptide

A short chain of amino acids that can be matched with a spectrum to form an identification.  This usage of peptide implies distinctness, i.e. that it may have been identified to many different spectra in the input data, but it only counts as one peptide.  A unique peptide, on the other hand, is one that belongs to only one protein in the FASTA database.

 

Peptide Group (metapeptide)

A group of results that share the same set of proteins.  These groups are formed to make the analysis more presentable.

Protein

A unique accession string that identifies a large chain of amino acids with biological meaning in a FASTA database.

Protein Group (metaprotein)

A group of proteins that share the same set of results.  They are indiscernible from each other based on available evidence.

Rank

A number assigned to a result to establish its relative ordering to other results for a given spectrum.  Lower ranks are better.

Result

One or more identifications that matched to a spectrum with the same score and thus produced the same FDRs.  When a result has more than one identification, all of them are presented together to emphasize the ambiguity of the result.

Sequence

A sequence simply refers to a peptide (see Peptide).

SID

A sequence identification number for distinguishing the different members of a given protein or peptide group.

Source Group

A canonical and hierarchical assignment of one or more input files to cause them to be analyzed separately (as well as with other groups).

Spectrum (Spectra)

A centroided list of mass-to-charge peaks from a tandem mass spectrum with an assigned charge state.
