ProteoWizard Project Overview

Directory Layout

  • pwiz
    • bin-linux [tested Linux binaries]
    • bin-windows [tested Windows binaries]
    • build [libraries, executables, and packages are built here]
    • doc [documentation]
    • libraries [3rd party library archives]
    • package [scripts for creating distributions]
    • src [source files]
    • test [regression tests]

Projects

Here is an outline of the various ProteoWizard projects, organized by dependency level.   There may be dependencies within a given level, but there should never be any up-level dependencies.   Unless otherwise noted, all projects are cross-platform.   Each project's source files are contained in the subdirectory of pwiz/src of the same name.

level 0 (base):

  • util:  Utility classes (Base64 encoding, sha1 hashing, XML parsing, 2D drawing, unit testing)
  • math:  Mathematics classes (linear algebra, statistics, special mathematical functions)

level 1 (data):

  • rawfile:  C++ interface to XCalibur COM object XRawfile2 (Windows/MSVC only).
  • msrun:  Abstraction interface for accessing data from either RAW or mzXML files
  • data:  Library containing classes for handling FT transient data, complex frequency data, MS1 peak data.
  • proteome:  Library containing classes for handling chemical formulas, peptides, and IPI FASTA database parsing

level 2 (analysis):

  • peaks:  Library of routines for frequency-domain peak detection.
  • calibration:  Library of routines for scan-level database calibration via EM algorithm.

level 3 (tools):

  • raw2xml:  Command-line tool to convert RAW files to mzXML
  • mspp:  Command-line tool for extracting data from RAW/mzXML files.
  • msprefix:  Command-line tool for preprocessing of RAW/mzXML data.
  • tools:  Command-line tools for processing transient data.

Code Conventions

A code module consists of an interface (Foo.hpp), implementation (Foo.cpp), and a unit test (FooTest.cpp).   The interface should be self-documenting, with optional inclusion of comment markup for automated documentation tools (e.g. Doxygen).   The unit test serves two purposes:
  1. To exercise the module's interface and validate its behavior independent of other modules.
  2. To document the intended usage of the code module.
Clients of a code module should never need to look at the implementation for questions about usage.
